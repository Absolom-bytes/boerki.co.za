# Boerki Platform - Terms and Conditions Code Index

This document outlines the software capabilities, data structures, and access control mechanics implemented in the Boerki application that MUST be disclosed or governed by the Terms and Conditions (T&C). It serves as a technical bridge for the legal/ecosystem design phase.

## 1. Data Collection and Storage (Firestore)
The following user-generated content and Personally Identifiable Information (PII) is stored persistently on Firebase/Google Cloud:
- **User Profiles (`/users/{userId}`)**: Email addresses, display names, profile pictures (URLs), and biography text.
- **Direct Messaging (`/chats` & `/chats/{chatId}/messages`)**: Private, inter-profile messaging data is stored and time-stamped.
- **Forum Inputs (`/branches`, `/threads`, `/comments`)**: All posts, descriptions, and comments generated by users are actively cached and stored indefinitely unless deleted.
- **Connections (`followers` & `following`)**: Social graphs mapping user connections.
- **Contact Forms (`/contacts`)**: Information submitted via contact or support forms.

*T&C Requirement: Explicit consent for cloud storage of PII, chat data (including that staff/admins *could* theoretically inspect it if escalated, though rules restrict general read), and public forum posts. Clarify data retention policies.*

## 2. Authentication & Persistent Sessions
- **Methods**: Users can authenticate via Google Single Sign-On (OAuth) or Email/Password.
- **Session Persistence**: The application manages user sessions via Firebase Auth, using local caching to keep users logged in.
- **Password Reset**: Users can trigger automated password reset emails.

*T&C Requirement: Statement regarding third-party auth providers (Google) and usage of local storage/cookies for session persistence.*

## 3. Administrator Privileges & Moderation Capabilities
The application includes a secure `/admin` dashboard ("Admin Konsole") bound by strict Firebase Security Rules (`isAdmin()` check). Admins have the ability to:
- **Global Read Access**: Traverse, read, and aggregate all database objects (users, threads, chats).
- **User Account Actions (`status` field)**: Modify user accounts by setting them to `active`, `suspended`, or `archived`.
- **Warnings System (`warnings` field)**: Issue numerical warnings to accounts for policy violations.
- **Content Moderation**: Delete or modify threads/comments in extreme circumstances, or suspend the authors.

*T&C Requirement: Clear definitions of "Acceptable Use". Users must agree that administrators have unilateral rights to issue warnings, throttle access, suspend, or permanently terminate accounts for violations without prior notice.*

## 4. Shared Community Spaces (Phase: Foundation to FET)
- Forums (Branches) are meant for educational scaling from Foundation to FET phases.
- Users create branches and self-manage them (acting as Branch Admins via the `adminIds` array).
- Threads and comments within branches are globally readable by authenticated users in the preview environment.

*T&C Requirement: Disclaimer on user-generated content liability. Boerki is not strictly liable for the content posted by branch creators or end users, though moderation policies apply.*

## 5. System Limitations & SLAs
- The platform relies on Firebase free/enterprise quotas for operations.
- Inter-profile messaging relies on real-time WebSockets (Firestore `onSnapshot`).
- Excessive traffic may trigger rate limits (e.g. Firebase `auth/too-many-requests`).

*T&C Requirement: "As Is" software clause. Boerki does not guarantee 100% uptime and is not liable for data loss or outages caused by upstream provider quotas or network issues.*
